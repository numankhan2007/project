=== SLIDE 1: PROJECT OVERVIEW ===
• Project Name: UNIMART
• Purpose: A secure, closed-ecosystem marketplace for university students to buy, sell, and trade items seamlessly.
• Target Users: Verified University Students.
• Core Idea: Eliminating scams and outsider interference on campus by locking profiles to an "Official Master Registry." Every transaction is strictly student-to-student and finalized with an OTP handshake mechanism.


=== SLIDE 2: TECH STACK ===
Frontend:
• Framework: React 18 with Vite
• Styling: TailwindCSS (Utility-first CSS)
• UI/UX Enhancements: Framer Motion (Animations), Lucide React (Icons)
• Routing & State: React Router DOM, React Context API
• HTTP Client: Axios

Backend:
• Framework: FastAPI (Asynchronous Python API framework)
• ORM & Migrations: SQLAlchemy, Alembic
• Background Tasks: APScheduler

Database & Infrastructure:
• Primary Database: PostgreSQL
• Cache & Rate Limiting: Redis (OTP storage and counters)
• Mail Service: SMTP (Python's native smtplib for local dev, previously SendGrid)

=== SLIDE 4: FRONTEND ANALYSIS ===
• Pages: App features robust routing wrapping pages like Login, Register, Dashboard, ProductPage, Orders, and ChatPage inside a ProtectedRoute component.
• Components: Employs a segmented component architecture (layout, common, product, chat). Forms use a custom Input component with dynamic error rendering.
• API Calls: Centralized API instances with Axios interceptors managing token injection and responses contextually inside `src/services/`.
• State Management: Comprehensive React Context providers (AuthContext, ThemeContext, OrderContext, ChatContext) eliminate aggressive prop-drilling.
• UI Features: Heavy use of Framer Motion for mounting/unmounting transitions. Intelligent loaders, skeletons, and Toast notifications ensure constant user feedback.


=== SLIDE 5: BACKEND ANALYSIS ===
• FastAPI Setup: High performance Async views. App lifecycle utilizes `lifespan` handlers to start database seeding (`official_records`) and background tasks seamlessly.
• Routers: Endpoints isolated into distinct files: `/auth` (Identity), `/products` (Inventory), `/orders` (Transactions), `/otp` (Verification), and `/admin` (Management).
• Authentication System: Utilizes JWT (JSON Web Tokens) generated after verifying hashed passwords (passlib/bcrypt). Custom dependency `get_current_user` extracts and decodes the JWT for secured routes.
• OTP Flow: Built heavily on Redis. OTPs are stored with TTLs (Time-To-Live). DB is spared from bloated, temporary OTP codes.
• Middleware: Global CORS middleware restricts access specifically to predefined frontend origins (localhost variants). Global exception catchers prevent application crashing on Redis/Postgres drops.


=== SLIDE 6: DATABASE DESIGN ===
The database implements a "Phase-based" integrity level design.

1. `official_records` (Read-Only Master)
   • Fields: register_number (PK), full_name, university, college, department, official_email
   • Purpose: Immutable root-of-truth. Students cannot register unless they exist here.

2. `user_profiles` (Read/Write)
   • Fields: register_number (PK/FK), username, hashed_password, personal_mail_id, phone_number
   • Purpose: Extends official data with app-specific mutable credentials.

3. `products`
   • Fields: id, seller_register_number(FK), title, price, image_urls (JSON API), product_status
   • Purpose: Stores active inventory, transitioning strictly from AVAILABLE -> RESERVED -> SOLD_OUT.

4. `orders` & `chat_messages`
   • Fields: id, product_id, buyer/seller register numbers, order_status
   • Purpose: Links a buyer, seller, and product. Chat operates under the umbrella of an Order ID.


=== SLIDE 7: API FLOW (VERY IMPORTANT) ===
• Register Flow: 
  Frontend (Reg No) -> Backend matches `official_records` -> Triggers OTP generation to official email via Redis -> Frontend submits OTP -> Backend flags email as verified -> Frontend submits Form -> Backend creates UserProfile & returns JWT.

• Login Flow: 
  Frontend -> `/auth/login` (accepts Username or Reg No) -> Checks bcrypt password hash -> Returns JWT.

• Product Upload Flow: 
  Frontend POST `/products` -> `get_current_user` verifies JWT -> Product added to database -> Available on market.

• OTP Transaction Flow:
  Buyer creates order -> Order goes PENDING -> Seller confirms -> "Initiate Delivery" generating Redis OTP -> Backend emails OTP to Buyer -> Physical meetup: Buyer inspects product -> hands OTP to Seller -> Seller enters OTP -> Order COMPLETED.


=== SLIDE 8: AUTHENTICATION SYSTEM ===
• JWT Usage: Security revolves around stateless Bearer tokens generated via PyJWT upon login or successful signup.
• Protected Routes: Backend endpoints inject `Depends(get_current_user)`. Frontend verifies authentication state locally via `AuthContext`.
• Admin Role System: Isolated endpoint environment (`/admin/`). Instead of JWT, it utilizes an environment variable (`ADMIN_KEY`) passed via the `X-Admin-Key` header.


=== SLIDE 9: OTP SYSTEM ===
• OTP Generation: Highly secure cryptographically safe library (`secrets.randbelow`) utilized over standard pseudo-random generators.
• Storage Strategy: OTPs NEVER hit the main PostgreSQL database actively. They are stored natively in Redis with a 600-second (10 min) memory expiry (TTL).
• Brute-Force Protection: Built-in Redis attempt counters (`otp_attempts:{email}`). The OTP is aggressively invalidated after 5 failed tries.
• Email Delivery: Dispatched natively asynchronously using Python's `smtplib` formatted via `MIMEMultipart` HTML blocks, drastically decreasing external API latencies.

=== SLIDE 12: ISSUES & RISKS ===
• Hardcoded Admin Risks: The admin credentials (`ADMIN_USERNAME` & `ADMIN_PASSWORD`) bypass database hashing entirely and sit as plaintext in `.env`.
• Image Handling: Images are stored merely as JSON strings array representations natively in `image_urls`. If base64 strings are submitted instead of CDN links, it heavily bloats the PostgreSQL rows leading to severe degradation.
• Scalability Risk: Relying entirely on HTTP requests for real-time `ChatMessage` architecture. High-frequency messaging will overload the PostgreSQL pool without WebSockets.
• SMTP Thread Blocking: Python's `smtplib` process can occasionally block event loops if external mail servers (like Gmail) throttle the connection.


=== SLIDE 13: IMPROVEMENTS ===
• Architecture: Upgrade the intra-transaction chat to utilize FastAPI `WebSockets`, enabling bidirectional asynchronous messaging.
• Optimizations: Introduce a Media Bucket Service (AWS S3, Cloudinary) to handle file uploads natively. The backend should generate presigned upload URLs rather than handling byte buffering.
• UI Improvements: Allow users multiple image uploads rendered as a swipeable gallery carousels on the product page.
• Scalability: Integrate Celery/Redis for robust background task queueing (specifically decoupling email dispatch workflows completely from the main thread).


=== SLIDE 14: FINAL SUMMARY ===
• Overall System Understanding: UNIMART represents a highly structured, defensively-programmed marketplace. It masterfully solves the "trust" issue inherent in second-hand trading platforms by strictly binding accounts to immutable institution-owned registries.
• Strengths: Phenomenal data normalization and segregation logic between `official_records` and mutable `user_profiles`. Best-in-class, Redis-powered temporary OTP architecture limiting attack vectors. UX is profoundly responsive.
• Weaknesses: Lacks an integrated cloud-storage pipeline for product media. The static Admin Access-Key format prevents multiple tiered-moderator accounts.
• Readiness for Deployment: The project is exceptionally capable for Beta deployment within an isolated University Sandbox. Transitioning the `.env` variables safely to a cloud provider and injecting an S3 bucket finishes its production readiness.
